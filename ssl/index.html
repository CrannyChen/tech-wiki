根憑證（Root CA Certificate）Root Certificate Authority
本質 自簽憑證（Self-Signed）：簽發者（Issuer）= 主體（Subject），沒有上層 CA，信任鏈的起點
儲存位置 預裝於作業系統與瀏覽器的「信任根憑證清單」中
Windows：憑證管理員（certmgr.msc） macOS：Keychain Access Firefox：自己維護獨立清單 Android：系統設定 → 安全 → 憑證
有效期 10–25 年（部分甚至更長，例如 DigiCert Global Root CA 有效期至 2031 年）
私鑰保管 離線（Air-gapped）硬體安全模組（HSM），極少動用，通常只在簽發或更新中繼憑證時才啟用
知名 Root CA DigiCert Global Root CA ISRG Root X1（Let's Encrypt） GlobalSign Root CA Sectigo AAA Google Trust Services
為何重要 瀏覽器信任某網站，最終依據是「信任這張根憑證」。若根 CA 私鑰外洩，整條信任體系全面崩潰，後果極為嚴重
撤銷困難 根憑證已預裝在數十億裝置中，一旦需要撤銷（Revoke），必須透過作業系統 / 瀏覽器的安全更新強制推送，耗時極長
🔍 如何查看：macOS → Keychain Access → 分類選「憑證」→ 左側選「系統根憑證」，可以看到 200+ 張預裝根憑證，每張都是你瀏覽器隱式信任的起點。
🔑
中繼憑證（Intermediate CA Certificate）Intermediate Certificate Authority
存在原因 根 CA 若直接簽發網站憑證，私鑰就必須保持線上狀態 → 風險極高。
改由中繼 CA 在線上運作，負責日常簽發工作，根 CA 私鑰維持離線
撤銷隔離 中繼 CA 若發生問題（被入侵、錯誤簽發），只需撤銷這張中繼憑證，不影響根 CA 的信任地位，也不需要更新所有裝置的根憑證清單
鏈的層數 可以有 1 到 2 張中繼憑證，視 CA 的業務規模與安全架構而定：
3 層：Root → Intermediate → Leaf（最常見） 4 層：Root → Int-1 → Int-2 → Leaf（大型 CA 常見）
有效期 通常 3–10 年
傳送方式 在 TLS 握手時，由伺服器主動傳送給瀏覽器（不像根憑證是本地預裝的）
設定錯誤 如果伺服器未傳送中繼憑證，各瀏覽器的行為差異很大：
Chrome / Edge
內建 AIA Fetching 機制，會自動讀取網站憑證裡的 Authority Information Access 欄位，主動去 CA 的 URL 下載缺少的中繼憑證，使用者看不到錯誤
自動補齊
Safari（macOS / iOS）
不執行 AIA Fetching，完全依賴伺服器主動傳送完整憑證鏈。若中繼憑證缺少，直接顯示「此伺服器的憑證鏈不完整」，使用者無法繼續連線
直接報錯
Firefox
維護自己的憑證快取，若過去訪問過使用相同中繼 CA 的網站，可能從快取補齊；但首次訪問時行為類似 Safari，不保證自動修復
視快取而定
⚠️ Nginx / Apache 設定重點：憑證檔案必須將中繼憑證串接在網站憑證後面，形成完整的 Chain File。
格式：cat domain.crt intermediate.crt > fullchain.crt
Let's Encrypt 自動產生的 fullchain.pem 已包含完整憑證鏈。
💡 測試建議：部署後除了用 Chrome 確認，務必也用 Safari（或 iPhone）測試一次。Chrome 自動補齊中繼憑證的特性，會讓你誤以為設定正確，但 Safari 使用者可能已經看到憑證錯誤。也可以用 openssl s_client -connect example.com:443 指令查看伺服器實際傳送的憑證鏈是否完整。

TLS 握手時憑證如何被使用TLS Handshake
1
Client Hello
瀏覽器發送支援的 TLS 版本（1.2 / 1.3）、加密套件清單（Cipher Suites）、亂數（Random Bytes）
2
Server Hello + 憑證鏈傳送
伺服器選定加密套件，並傳送：網站憑證 + 一張（或多張）中繼憑證。根憑證不需傳送，因為瀏覽器本地已有
3
瀏覽器驗證憑證鏈
逐層往上驗簽：
網站憑證的簽章 → 是否由中繼 CA 的公鑰可驗證？
中繼憑證的簽章 → 是否由根 CA 的公鑰可驗證？
根憑證 → 是否在本地信任清單中？
4
驗證網域 + 有效期
SAN 欄位中的網域是否符合你正在連線的主機名稱？Not Before / Not After 是否在有效範圍內？
5
確認撤銷狀態（OCSP）
向 CA 的 OCSP Responder 查詢憑證是否已被撤銷（Revoked）。現代瀏覽器多使用 OCSP Stapling 由伺服器預先附帶查詢結果
6
金鑰協商（Key Exchange）
TLS 1.3 使用 ECDHE，瀏覽器與伺服器各自產生暫時金鑰對，在不傳輸私鑰的情況下協商出相同的 Session Key
7
對稱加密通訊開始
之後所有 HTTP 請求與回應均以 Session Key 對稱加密傳輸（AES-GCM 等），憑證的工作到此結束
💡 憑證在 TLS 中的角色：僅用於「身份驗證與金鑰協商」階段，實際資料傳輸使用的是對稱加密，憑證本身不加解密任何網頁內容。

① 出廠預裝：你的電腦為什麼「天生」就信任 Google？
根憑證清單（Root Store）
你買一台新 MacBook，完全沒有瀏覽過任何網站，但打開瀏覽器連到 https://google.com 時，瀏覽器立刻顯示安全的鎖頭圖示。這是因為 Apple 出廠時就已在 macOS 裡預裝了一份「信任根憑證清單（Root Store）」，裡面有 200+ 張根憑證。
Windows
Microsoft Root Program 維護，存於系統登錄檔
macOS / iOS
Apple Root Program 維護，存於 Keychain
Firefox
Mozilla 自己維護，不使用系統清單
這份清單由作業系統廠商嚴格審核，CA 機構必須通過稽核才能被加入。
比喻：就像你手機出廠就存好了 119（消防）、110（警察）的電話號碼，製造商已先幫你驗證好這些號碼的真實性。
🔐
② 根憑證裡面裝了什麼？
公鑰 + 自簽簽章
根憑證本質上是一個檔案，裡面最重要的東西只有兩個：
Root CA 的公鑰
用來驗證「這個 CA 簽發的憑證」是否真實。就像一枚蓋章的印章樣本，讓你能比對真假
自簽簽章
Root CA 用自己的私鑰對自己簽章（自己背書自己），因為沒有上層 CA 可以替它背書
根憑證的對應私鑰則被 CA 機構存在一台完全斷網的電腦（Air-gapped），鎖在實體保險庫裡，每次動用都需要多人同時在場的儀式性程序，並全程錄影。
為什麼要這麼謹慎？根 CA 私鑰就是整個信任體系的「萬能鑰匙」。若私鑰外洩，攻擊者可以偽造任何網站的憑證，瀏覽器完全無法察覺。2011 年荷蘭 CA 機構 DigiNotar 遭入侵即為歷史慘案，最終宣告倒閉。
✍️
③「簽發」這個動作是什麼意思？
數位簽章的核心機制
「Root CA 簽發中繼憑證」這句話背後的數學操作：
中繼憑證內容 → 雜湊 → 用 Root CA 私鑰加密 → 簽章（Signature）
這個簽章被附加在中繼憑證裡面。之後任何人只要拿到中繼憑證，都可以用 Root CA 的公鑰（就是存在你電腦裡的那個）去解密簽章，再比對憑證內容的雜湊，確認憑證沒有被竄改、確實出自這個 Root CA。
  • 🔒私鑰加密 → 只有 Root CA 能做（不可偽造）
  • 🔑公鑰解密 → 任何人都能驗證（公開透明）
  • 🔍雜湊比對 → 確認內容沒被竄改（防篡改）
關鍵洞見：簽章讓「信任」可以被傳遞。Root CA 信任中繼 CA，所以它的簽章讓中繼憑證被信任；中繼 CA 再以同樣方式讓網站憑證被信任。信任沿著鏈條流動。
🌐
④ 你連到 https://google.com 時，瀏覽器做了什麼？
即時本地驗證流程
Google 伺服器傳來憑證鏈（不含根憑證）：
google.com 網站憑證 + GTS CA 1C3（中繼） ← 伺服器傳這兩張
瀏覽器收到後，在你電腦本地做這些事（完全不需要連線）：
  • ①拿網站憑證的 Issuer 欄位，找到對應的中繼憑證
  • ②用中繼憑證的公鑰，驗證網站憑證上的簽章 → 確認網站憑證合法
  • ③拿中繼憑證的 Issuer，找到本地根憑證清單
  • ④用根憑證的公鑰，驗證中繼憑證上的簽章 → 確認中繼憑證合法
  • ⑤確認根憑證在本地信任清單中 → 信任成立 ✅
  • ⑥確認網域名稱符合（SAN 欄位）、有效期未過
重點：根憑證從頭到尾都不需要傳送，也不需要連線查詢，因為它一直靜靜地存在你電腦裡。整個驗證過程在毫秒內完成。
⚠️
⑤ 如果驗證失敗，會發生什麼？
信任斷鏈的各種情況
常見的 HTTPS 錯誤及原因：
  • ⚠️ERR_CERT_AUTHORITY_INVALID
    憑證簽發者不在本地根憑證清單 → 常見於自簽憑證、公司內網 CA、或企業 SSL 攔截設備
  • ⚠️ERR_CERT_DATE_INVALID
    憑證已過期，或電腦時間設定錯誤（瀏覽器以本地時間判斷）
  • ⚠️ERR_CERT_COMMON_NAME_INVALID
    憑證的 SAN 網域與你連線的網域不符
  • ⚠️憑證鏈不完整
    伺服器沒有傳中繼憑證，瀏覽器找不到中間那層，信任鏈斷掉
另一種特殊情況：企業 SSL 攔截（Corporate MITM）—— 公司 IT 在你電腦裡安裝了公司自己的根憑證，讓防火牆設備能解密並重新加密所有 HTTPS 流量，瀏覽器看到的是「合法的」公司憑證，但實際上流量被中間解開過了。
核心假設：整個機制假設「你電腦裡的根憑證清單是可信的」。如果惡意程式偷偷安裝假的根憑證，它就能偽裝任何網站而不被察覺，這也是防止惡意軟體的重要原因之一。

常見問題
每個 HTTPS 都有這三種憑證嗎？ 邏輯上一定有根憑證與網站憑證，中間層數則不固定：
2 層：Root → Leaf（罕見，部分企業內網 / 自簽環境）
3 層：Root → Intermediate → Leaf（最常見）
4 層：Root → Int-1 → Int-2 → Leaf（大型商業 CA）
如何查看某網站的憑證鏈？ Chrome：點網址列鎖頭 → 連線是安全的 → 憑證有效 → 詳細資訊
命令列：openssl s_client -connect example.com:443 -showcerts
憑證過期了怎麼辦？ 重新向 CA 申請（自動化工具如 Certbot 可搭配 cron job 在到期前自動更新）。過期憑證瀏覽器會顯示警告，無法正常連線
HTTP 與 HTTPS 的差別？ HTTP 無任何憑證，傳輸明文，任何中間人可攔截讀取或竄改。HTTPS = HTTP over TLS，透過以上憑證機制確保身份真實性、內容加密與完整性

如何建立憑證Certificate Creation
正式憑證申請
自簽憑證（Self-Signed）
1
產生私鑰（Private Key）
在伺服器本地產生，私鑰永遠不離開伺服器。
# RSA 2048-bit（傳統）
openssl genrsa -out server.key 2048

# ECDSA P-256（現代，更快更小）
openssl ecparam -name prime256v1 -genkey -noout -out server.key
2
產生 CSR（Certificate Signing Request）
CSR 是送給 CA 的「申請書」，包含你的公鑰和網域資訊，不含私鑰，可以安全地傳送。
openssl req -new -key server.key -out server.csr \
  -subj "/C=TW/ST=Taipei/O=MyCompany/CN=www.example.com"

# 查看 CSR 內容確認
openssl req -text -noout -in server.csr
CSR 包含：公鑰、網域名稱（CN）、組織資訊。不包含私鑰。
3
選擇 CA 並提交 CSR
依需求選擇 CA：
🆓 Let's Encrypt（免費）
✅ 完全免費，自動化
✅ DV 驗證，幾分鐘完成
✅ Certbot 自動申請 + 續期
⚠️ 有效期 90 天，需自動更新
❌ 不提供 OV / EV
💰 商業 CA（付費）
✅ DV / OV / EV 都提供
✅ 有效期 1 年
✅ 客服支援、保險保障
⚠️ OV/EV 需提供公司文件
💰 DigiCert、Sectigo、GlobalSign
4
通過網域驗證（DV）
CA 需確認你真的控制這個網域，有三種方式：
# 方式 1：HTTP 驗證（最常見）
# CA 要求你在網站放一個特定檔案
http://example.com/.well-known/acme-challenge/abc123

# 方式 2：DNS 驗證（萬用字元憑證必須用這個）
# 在 DNS 加一筆 TXT 記錄
_acme-challenge.example.com  TXT  "隨機驗證字串"

# 方式 3：Email 驗證
# 寄信到 admin@example.com 或 webmaster@example.com
5
取得憑證並部署到伺服器
CA 審核通過後給你憑證檔案，需串接中繼憑證再設定到 Nginx / Apache。
# 串接成完整憑證鏈
cat domain.crt intermediate.crt > fullchain.crt

# Nginx 設定
ssl_certificate     /etc/ssl/fullchain.crt;
ssl_certificate_key /etc/ssl/server.key;
推薦：使用 Certbot 全自動申請（Let's Encrypt）
# 安裝 Certbot（Ubuntu）
sudo apt install certbot python3-certbot-nginx

# 一行搞定申請 + Nginx 設定
sudo certbot --nginx -d example.com -d www.example.com

# 自動續期測試
sudo certbot renew --dry-run
Certbot 會自動：產生 CSR、完成 HTTP 驗證、下載憑證 + 中繼憑證、修改 Nginx 設定、定期自動續期。基本上不需要手動跑任何 openssl 指令。
自簽憑證（Self-Signed Certificate）：你自己當 CA，自己簽自己的憑證。瀏覽器因為不認識這個 CA，一定會顯示警告。適合本機開發、公司內網、Docker 容器間通訊。
1
一行產生私鑰 + 自簽憑證
openssl req -x509 -newkey rsa:2048 \
  -keyout key.pem -out cert.pem \
  -days 365 -nodes \
  -subj "/CN=localhost"
-x509 直接產生憑證（跳過 CSR）  -nodes 私鑰不加密  -days 365 有效期一年
2
Nginx 設定
ssl_certificate     /path/to/cert.pem;
ssl_certificate_key /path/to/key.pem;
限制：Chrome 58+ 要求憑證必須有 SAN 欄位，只有 CN 的憑證會報 ERR_CERT_COMMON_NAME_INVALID。若需要瀏覽器測試，請改用「含 SAN」做法。
1
建立設定檔（含 SAN）
# openssl.cnf
[req]
distinguished_name = req_distinguished_name
x509_extensions    = v3_req
prompt             = no

[req_distinguished_name]
CN = localhost

[v3_req]
subjectAltName = @alt_names

[alt_names]
DNS.1 = localhost
DNS.2 = *.localhost
IP.1  = 127.0.0.1
IP.2  = ::1
2
產生憑證
openssl req -x509 -newkey rsa:2048 \
  -keyout key.pem -out cert.pem \
  -days 365 -nodes \
  -config openssl.cnf
3
讓瀏覽器信任（本機開發用）
# macOS
sudo security add-trusted-cert -d -r trustRoot \
  -k /Library/Keychains/System.keychain cert.pem

# Windows PowerShell（管理員）
Import-Certificate -FilePath cert.pem \
  -CertStoreLocation Cert:\LocalMachine\Root
推薦工具 mkcert：本機開發最佳解法，自動建立本機 CA、產生含 SAN 的憑證、並自動加入系統信任。
brew install mkcert && mkcert -install && mkcert localhost 127.0.0.1
適合公司內網：建立自己的根 CA，讓所有內部伺服器的憑證都由它簽發。只需把根 CA 推送到員工電腦，就能消除所有內部憑證警告。
1
建立根 CA 私鑰與憑證
# 產生根 CA 私鑰（加密保護）
openssl genrsa -aes256 -out ca.key 4096

# 產生根 CA 自簽憑證（有效 10 年）
openssl req -x509 -new -nodes -key ca.key \
  -sha256 -days 3650 -out ca.crt \
  -subj "/C=TW/O=MyOrg/CN=MyOrg Root CA"
2
為每台伺服器產生私鑰 + CSR
openssl genrsa -out server.key 2048

openssl req -new -key server.key -out server.csr \
  -subj "/CN=internal.myorg.com" \
  -addext "subjectAltName=DNS:internal.myorg.com,DNS:*.myorg.com"
3
用根 CA 簽發伺服器憑證
openssl x509 -req -in server.csr \
  -CA ca.crt -CAkey ca.key \
  -CAcreateserial \
  -out server.crt -days 365 -sha256
4
把根 CA 推送到員工電腦
# macOS 手動
sudo security add-trusted-cert -d -r trustRoot \
  -k /Library/Keychains/System.keychain ca.crt

# Windows GPO（群組原則，大規模部署）
# certlm.msc → 受信任根憑證授權單位 → 匯入
推送後，所有由這個 CA 簽發的內部憑證，員工瀏覽器都會顯示安全鎖頭。
安全提醒：ca.key 是你的根 CA 私鑰，務必妥善保管並加密存放。若外洩，所有內部憑證的信任基礎即崩潰。
Spring Boot 額外補充：若要在 Spring Boot 使用自簽憑證，可轉成 PKCS12 格式：
openssl pkcs12 -export -in server.crt -inkey server.key -out keystore.p12 -name server
再在 application.properties 設定：
server.ssl.key-store=classpath:keystore.p12
server.ssl.key-store-type=PKCS12
server.ssl.key-store-password=yourpassword

各階段憑證的簽發時機When Each Certificate Is Issued
核心規律：簽發頻率與憑證在信任鏈的位置成反比 —— 越靠近根的越少動，越靠近使用者的越頻繁換。這是分層設計的目的，讓高風險的頻繁操作遠離根 CA 私鑰。
Root CA
根憑證
幾乎一次性
CA 機構成立時建立，有效期 10–25 年，之後幾乎不動
Root CA Rollover：到期前數年提前建立下一代根憑證，推送至各作業系統，新舊重疊確保過渡期無縫銜接
私鑰外洩（極罕見）：立即廢棄舊根憑證，啟動緊急程序重建整條信任鏈，影響範圍極大（參考 DigiNotar 事件）
Intermediate CA
中繼憑證
每 3–10 年
正常輪換：到期前由根 CA 重新簽發，通常提前 6–12 個月作業
遭入侵或錯誤簽發：立即撤銷（CRL/OCSP），重新簽發新的中繼憑證，不影響根 CA
業務擴展：針對不同地區、產品線建立額外的中繼 CA，分散風險
End-Entity
網站憑證
90 天–1 年
第一次架設 HTTPS：申請第一張憑證
到期前續期：Let's Encrypt 90 天、商業 CA 1 年。Certbot 可設定自動在到期前 30 天更新
網域變更：新增或移除 SAN 網域時，舊憑證無法修改，需重新申請
私鑰外洩或疑似入侵：立即撤銷並重新產生私鑰 + 申請新憑證，不能沿用舊私鑰
更換或重建伺服器：私鑰不應跨機器搬移，在新機器重新產生即可

② 指令列（最準確）
1
查看完整憑證鏈（含中繼憑證）
最重要的指令，可確認伺服器實際傳送了哪些憑證，用來診斷 Safari 憑證鏈不完整問題。
openssl s_client -connect example.com:443 -showcerts

# 輸出結果中，每個 BEGIN CERTIFICATE ... END CERTIFICATE 是一張憑證
# 正常應看到 2 張（網站憑證 + 中繼憑證）
# 只看到 1 張 → 伺服器沒傳中繼憑證，Safari 會報錯
2
快速查看憑證基本資訊
一行看到網域、發行者、有效期起訖。
echo | openssl s_client -connect example.com:443 2>/dev/null   | openssl x509 -noout -subject -issuer -dates

# 範例輸出：
# subject=CN=example.com
# issuer=CN=GTS CA 1C3, O=Google Trust Services LLC
# notBefore=Mar  1 00:00:00 2025 GMT
# notAfter =May 30 00:00:00 2025 GMT
3
只查到期日
echo | openssl s_client -connect example.com:443 2>/dev/null   | openssl x509 -noout -enddate

# notAfter=May 30 00:00:00 2025 GMT
4
驗證憑證鏈完整性（模擬 Safari 行為）
加上 -verify_return_error 讓 openssl 嚴格驗證，不自動補中繼憑證，最接近 Safari 的行為。
openssl s_client -connect example.com:443   -verify_return_error   -partial_chain 2>&1 | grep -E "Verify|error|chain"

# 正常：Verify return code: 0 (ok)
# 異常：verify error:num=20:unable to get local issuer certificate
