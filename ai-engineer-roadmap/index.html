Attention 機制

self-attention & multi-head

+
  • Self-Attention：Q、K、V 計算流程
    Query × Key → softmax → Value 加權求和；每個 token 同時「看」所有其他 token，計算彼此關聯強度
    公式：Attention(Q,K,V) = softmax(QKᵀ / √d_k) × V，除以 √d_k 防止梯度消散
  • Multi-Head Attention（多頭注意力）
    把 Attention 並行執行 h 次（每頭有獨立的 W_Q、W_K、W_V 矩陣），捕捉不同子空間的語義關係，最後 concat 再線性變換
  • Positional Encoding
    Transformer 本身無法感知順序，需加入位置資訊。原始論文用 sin/cos 函數；現代 LLM（Llama、Mistral）用 RoPE（旋轉位置編碼），支援更長 context 的外推
  • Feed-Forward Network（FFN）
    Attention 層後接兩層全連接網路 + 啟動函數（GELU / SiLU）；負責儲存「事實知識」；SwiGLU 是現代 LLM 的標準選擇
  • Layer Norm & Residual Connection
    殘差連接讓梯度穩定反傳；Pre-LN（現代，放 Attention 前）比 Post-LN（原始論文）訓練更穩定；RMSNorm 是 Llama 的選擇（更快）

Tokenization & Embedding

文字如何變成數字

+
  • BPE（Byte-Pair Encoding）
    GPT 系列使用的分詞演算法；反覆合併高頻位元組對建立詞彙表；中文通常 1 個字 = 1~2 token（英文約 4 字元 = 1 token）
  • Token 實用認知
    程式碼縮排、空格都算 token；中文 1M tokens ≈ 50萬字；token 數直接影響 API 成本和 context 用量，要養成估算習慣
  • Word Embedding vs Contextual Embedding
    Word2Vec 每個詞固定一個向量（「蘋果」水果 vs 公司 同一向量）；Transformer 每個詞的向量根據上下文動態計算，解決歧義問題
  • Context Window & KV Cache
    模型能處理的最大 token 數（GPT-4o 128K，Claude 200K）；KV Cache 快取歷史 Key/Value，避免每步重新計算，大幅提升推理速度

推理參數控制

Temperature / Top-p / Sampling

+
  • Temperature
    控制 logit 分佈的尖銳程度。0 → 永遠選最高機率 token（貪婪、確定性）；1 → 按原始機率；2 → 高度隨機。資料提取用 0，創意寫作用 1~1.2
  • Top-p（Nucleus Sampling）
    從累積機率 ≥ p 的最小 token 集合中採樣。Top-p=0.9 表示只考慮機率加總達 90% 的 token；比 Top-k 更動態自適應，是現代主流
  • Repetition Penalty / Frequency Penalty
    懲罰已出現的 token 讓模型避免重複；結構化輸出（JSON）設低一點避免格式被干擾；對話情境建議 1.1~1.3
  • Stop Sequence
    讓模型遇到特定字串時停止生成；常用 ["}", "\n\n", "```"]；控制輸出邊界避免無限生成
情境建議設定
程式碼生成Temp 0.1~0.3，Top-p 0.95
對話問答Temp 0.5~0.7，Top-p 0.9
創意寫作Temp 1.0~1.2，Top-p 0.95

核心提示技術

CoT / Few-shot / Structured Output

+
  • Zero-shot vs Few-shot
    Zero-shot 直接描述任務；Few-shot 提供 3~5 個範例讓模型學格式；範例品質比數量重要；範例要覆蓋邊界情況
  • Chain-of-Thought（CoT）
    「Let's think step by step」讓模型展示推理過程；數學/邏輯任務提升顯著；Self-Consistency：跑多次 CoT 取多數答案，進一步提升正確率
  • System Prompt 設計原則
    • 明確定義角色、任務範圍、輸出格式
    • 說「要做什麼」比說「不要做什麼」更有效
    • 複雜任務拆成有序步驟，用編號列出
    • 重要指令放開頭和結尾（Lost in Middle 問題）
  • Structured Output（JSON Mode / XML）
    OpenAI JSON Mode 保證合法 JSON；Claude 擅長 XML 標籤（<result>...</result>）；Instructor 套件讓輸出自動驗證成 Pydantic Schema
  • Lost in the Middle 效應
    Long context 模型容易忽略中間內容；重要資訊放最前或最後；RAG 時把最相關的 chunk 排在前面

Prompt Injection & 防護策略

+

    • 輸入長度限制：過長輸入通常有問題

Ingestion Pipeline

文件解析與切分策略

+
  • 文件解析工具選擇
    • pypdf / pdfplumber：純文字 PDF，輕量
    • unstructured：多格式統一解析（PDF/Word/HTML/Excel）
    • docling（IBM）：保留表格結構，OCR 品質佳
    • 掃描 PDF 需 OCR：tesseract 或 AWS Textract
  • Chunking 四大策略
    • Fixed-size：按字元數切，簡單但常切斷語義
    • Recursive Character：按段落→句子→字元遞迴，最推薦
    • Semantic：用 Embedding 相似度決定切點，品質最好但最慢
    • Hierarchical（Parent-Child）：小塊精準定位 + 大塊提供上下文
  • Chunk Size 與 Overlap 經驗值
    一般 Q&A：256~512 tokens；長文閱讀：512~1024；Overlap 設 10~15%（50~100 tokens）避免語義在邊界截斷；chunk 太小上下文不足，太大雜訊多
  • Metadata 提取與保存
    保存來源（檔名/頁碼/URL）、日期、章節標題、作者；後續可用 Metadata Filter 縮小搜尋範圍（只搜 2024 年的文件）

Retrieval 進階技術

Hybrid Search / Re-ranking / Query 轉換

+
  • Hybrid Search（向量 + 全文）
    語義搜尋召回「意思相近」；BM25 召回「關鍵詞完全匹配」；兩者互補。用 RRF（Reciprocal Rank Fusion）合併排名，公式：1/(k + rank_i)
  • Re-ranking（二階段精排）
    Bi-Encoder（向量搜尋）初排取 Top-50；Cross-Encoder 逐一計算 (query, doc) 相關性精排取 Top-5；精排模型：bge-reranker-v2-m3、Cohere Rerank
  • Query 轉換技術
    • Query Rewriting：LLM 改寫問題讓搜尋更清晰
    • HyDE（假設文件擴展）：先讓 LLM 假設答案，用答案向量搜尋
    • Multi-Query：一問題生成多個角度查詢，聯集後去重
    • Step-back Prompting：先問更抽象的父問題，再細化
  • Context 組裝策略
    • Stuff：把所有 chunk 塞進 prompt（簡單，token 限制）
    • Map-Reduce：各 chunk 單獨問再彙整（適合超長文件）
    • Refine：逐塊精煉答案（較慢但品質高）

Fine-tuning 方法全覽

LoRA / QLoRA / SFT / DPO

+
  • Full Fine-tuning vs LoRA vs QLoRA
    • Full：更新所有參數；效果最好；70B 模型需 140GB+ VRAM
    • LoRA：插入低秩矩陣（A×B），參數量降 100x，推理時合併不增加延遲
    • QLoRA：原始模型 4-bit 量化 + bf16 LoRA；8B 模型只需 6GB VRAM
  • SFT（Supervised Fine-Tuning）
    用（instruction, response）對訓練；讓模型學習特定風格 / 領域術語 / 輸出格式；資料格式：{"messages": [{"role":"user",...},{"role":"assistant",...}]}
  • DPO（Direct Preference Optimization）
    用（chosen, rejected）對訓練；比 RLHF 更簡單穩定；讓模型輸出更符合人類偏好；不需要獨立的 Reward Model
  • RAG vs Fine-tuning 選擇
    情境選擇
    知識會定期更新RAG
    需要引用來源出處RAG
    改變輸出風格/格式Fine-tuning
    學習特定領域術語Fine-tuning
    最佳效果兩者結合

RAG 評估框架

RAGAS & LLM-as-Judge

+
  • RAGAS 四大指標
    • Faithfulness（0~1）：答案是否完全基於 Context，防幻覺核心指標
    • Answer Relevancy（0~1）：答案是否真的回答了問題
    • Context Precision（0~1）：檢索到的 chunk 有多少真正有用
    • Context Recall（0~1）：應該找到的資訊有沒有都被召回
  • LLM-as-Judge 評估
    用 GPT-4 / Claude 對另一個模型輸出評分（1~5 分 + 理由）；設計評分 Rubric 保持一致性；成本低且可擴展，但評分本身有偏差需注意
  • Golden Dataset 建立
    手動標記 50~200 個（問題, 標準答案）對；每次 Prompt 或模型改版後自動跑 regression test；是衡量改進的客觀基準

Agent 架構與模式

ReAct / Plan-Execute / Memory

+
  • Agent Loop（感知→推理→行動→觀察）
    每個循環：接收環境輸入 → LLM 推理下一步 → 執行工具 → 觀察結果 → 重複直到任務完成或達到最大步數
  • ReAct Pattern（Reason + Act）
    交錯輸出 Thought → Action → Observation；可解釋每步推理過程；適合工具使用；輸出格式：
    • Thought: 我需要先查詢使用者的訂單資料
    • Action: get_order(user_id=123)
    • Observation: [訂單列表...]
    • Final Answer: 您有 3 筆訂單...
  • Tool / Function Calling 完整流程
    • ① 定義工具 JSON Schema（name / description / parameters）
    • ② LLM 決定呼叫哪個工具、帶哪些參數
    • ③ 後端執行實際邏輯（查 DB / 呼叫 API）
    • ④ 把結果塞回 messages 繼續對話
  • Memory 系統四層
    • Short-term（Working Memory）：對話 Context Window，有限
    • Episodic（情節記憶）：具體事件記錄，壓縮後存向量 DB
    • Semantic（語義記憶）：知識庫（RAG）
    • Procedural（程序記憶）：System Prompt 裡的固定指令
  • Planning 策略
    • Task Decomposition：LLM 先輸出步驟清單再執行
    • Plan-and-Execute：規劃和執行分離，規劃者 + 執行者兩個 LLM
    • ReWOO：預先生成所有工具呼叫計劃，避免重複推理

MCP — Model Context Protocol

Anthropic 開放標準 2024

+
  • MCP 是什麼？為什麼重要？
    讓 LLM 連接外部工具的統一標準協定（類似 USB 介面）；沒有 MCP 前，每個 AI 應用要各自實作工具整合；MCP 讓「寫一次 Server，所有 AI Host 都能用」
  • 三層架構
    • Host：Claude Desktop、IDE（Cursor、Windsurf）、自建應用
    • Client：每個 MCP 連接的管理者（在 Host 內）
    • Server：提供工具的服務（可本地或遠端）
  • 三種能力類型
    • Tools：可執行操作（查資料庫、呼叫 API、讀寫檔案）
    • Resources：可讀取資料（檔案內容、DB 記錄、API 回應）
    • Prompts：預定義 Prompt 模板（可帶參數）
  • 傳輸方式

推理引擎 & 量化技術

vLLM / TGI / llama.cpp

+
  • vLLM（最推薦生產選擇）
    PagedAttention：KV Cache 分頁管理，大幅減少記憶體碎片；Continuous Batching：動態合併請求，GPU 利用率遠高於靜態 Batching；Throughput 是原生 HuggingFace 的 10~24x
  • TGI（Text Generation Inference，HuggingFace）
    Flash Attention 2；Tensor Parallelism（多 GPU）；Token Streaming；Docker 映像即用；適合 HuggingFace 模型生態
  • 量化技術比較
    • GGUF（llama.cpp）：CPU 推理 / Apple Silicon；4-bit 量化 7B ≈ 4GB，13B ≈ 8GB
    • GPTQ：GPU INT4 量化；精確度損失小
    • AWQ（Activation-aware）：量化品質優於 GPTQ；vLLM 原生支援
    • FP8：NVIDIA H100/H200 原生支援；精確度損失最小
  • 部署方案選型
    方案適用情境
    API 直呼開發期 / 低量 / 快速上線
    vLLM + K8s高量生產，需要控制模型版本
    Ollama 本地開發測試 / 資料不能離機
    Bedrock / Azure OpenAI合規要求 / 企業採購合約

Streaming & Latency 優化

SSE / Prompt Caching / Speculative Decoding

+
  • Latency 指標
    • TTFT（Time to First Token）：使用者感知最關鍵，應 < 1s
    • TPOT（Time per Output Token）：每個 token 生成時間
    • E2E Latency：完整回應時間 = TTFT + (token數 × TPOT)
  • Prompt Caching（重要！）
    Claude 和 GPT-4o 支援 System Prompt 快取；相同前綴不重複計費計算；長 System Prompt 可省 90% token 費用；適合 RAG 的固定 Context
  • Speculative Decoding
    用小 Draft Model 預測多個 token，大 Target Model 批量驗證；成功率高時 2~3x 速度提升；vLLM 原生支援
  • SSE Streaming 實作（Spring Boot）
    SseEmitter（簡單）或 Flux<String>（WebFlux reactive）；前端用 EventSource / fetch + ReadableStream 接收；需設 Content-Type: text/event-stream

LLM Tracing & 監控

Langfuse / LangSmith / 成本追蹤

+
  • 需要追蹤的核心指標

  • Prompt 版本管理

Guardrails & Hallucination 防護

輸入輸出過濾

+
  • Hallucination 偵測與防護
    • RAG 引用來源：強制 LLM 只從 Context 回答，加「若 Context 沒有資訊請說不知道」
    • Faithfulness 驗證：第二個 LLM 核實答案是否符合 Context
    • 信心閾值：向量相似度低於閾值 → 直接回「我不知道」
    • Self-RAG：讓模型自我評估要不要繼續檢索
  • Input Guardrails
    • PII 偵測：用 Presidio（微軟開源）偵測並遮蔽姓名/電話/身份證
    • 有害意圖分類：用小型分類模型（DistilBERT）判斷輸入類別
    • 長度限制：超長輸入通常有攻擊意圖
    • Prompt Injection 偵測：關鍵詞過濾 + LLM 二次判斷
  • Output Guardrails
    • JSON Schema 驗證：輸出不符合格式就重試（最多 3 次）
    • 有害內容過濾：OpenAI Moderation API / Perspective API
    • PII 二次遮蔽：輸出前再過濾一次，防止 LLM 洩露訓練資料
