Mac 專屬

Nginx — Web Server 安裝設定

第四步 對外入口
Nginx 負責接收外部 HTTP/HTTPS 請求，然後反向代理（Reverse Proxy）到後端 Tomcat。

安裝 Nginx

安裝
bash
## 安裝 $ sudo apt install -y nginx ## 啟動並設定開機自啟 $ sudo systemctl start nginx $ sudo systemctl enable nginx ## 確認狀態 $ sudo systemctl status nginx ● nginx.service - A high performance web server Loaded: loaded (/lib/systemd/system/nginx.service; enabled) Active: active (running) ## 在瀏覽器輸入 VM 的 IP 應該看到 Nginx 歡迎頁 # http://192.168.50.130

Nginx 目錄結構

結構
/etc/nginx/ ├── nginx.conf 主設定檔（通常不直接改） ├── sites-available/ 設定檔存放目錄 │ ├── default 預設網站設定 │ └── myapp ← 你的自訂設定放這 ├── sites-enabled/ 啟用的設定（用 symlink 連結） │ └── myapp → ../sites-available/myapp └── conf.d/ 全域設定 /var/log/nginx/ ├── access.log 存取記錄 └── error.log 錯誤記錄

設定反向代理到 Tomcat

關鍵設定
/etc/nginx/sites-available/myapp
server { listen 80; server_name myapp.example.com; # 或直接用 IP # 存取 log access_log /var/log/nginx/myapp_access.log; error_log /var/log/nginx/myapp_error.log; # 反向代理到 Tomcat location / { proxy_pass http://127.0.0.1:8080; proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; proxy_set_header X-Forwarded-Proto $scheme; # 超時設定 proxy_connect_timeout 30s; proxy_send_timeout 60s; proxy_read_timeout 60s; } # 靜態檔案直接由 Nginx 處理（效能更好） location /static/ { root /opt/myapp/webroot; expires 30d; } }
bash — 啟用設定
# 建立 symlink 啟用設定 $ sudo ln -s /etc/nginx/sites-available/myapp /etc/nginx/sites-enabled/ # 刪掉預設設定（避免衝突） $ sudo rm /etc/nginx/sites-enabled/default # 測試設定語法 $ sudo nginx -t nginx: the configuration file /etc/nginx/nginx.conf syntax is ok nginx: configuration file /etc/nginx/nginx.conf test is successful # 重新載入設定（不中斷連線） $ sudo systemctl reload nginx

SSL 憑證掛載

第四步 HTTPS
SSL 憑證讓網站可以用 HTTPS 連線。正式環境用 CA 憑證，測試環境可以用 Let's Encrypt 或自簽憑證。

方法一：Let's Encrypt（免費，正式環境推薦）

免費
⚠️
需要有真實的網域名稱且 DNS 指向你的伺服器，VM 練習環境無法使用。
bash — Let's Encrypt
## 安裝 Certbot $ sudo apt install -y certbot python3-certbot-nginx ## 申請憑證（自動修改 Nginx 設定） $ sudo certbot --nginx -d myapp.example.com Congratulations! Your certificate and chain have been saved at: /etc/letsencrypt/live/myapp.example.com/fullchain.pem ## 憑證 90 天過期，設定自動更新 $ sudo certbot renew --dry-run # 測試更新 # Certbot 安裝後會自動加入 crontab 定期更新

方法二：自簽憑證（測試環境 / VM 練習）

測試
bash — 自簽憑證
## 建立目錄 $ sudo mkdir -p /etc/nginx/ssl ## 產生自簽憑證（有效期 365 天） $ sudo openssl req -x509 -nodes -days 365 -newkey rsa:2048 \ -keyout /etc/nginx/ssl/myapp.key \ -out /etc/nginx/ssl/myapp.crt \ -subj "/C=TW/ST=Taiwan/L=Taipei/O=MyOrg/CN=192.168.50.130" ## 確認憑證 $ openssl x509 -in /etc/nginx/ssl/myapp.crt -text -noout | grep "Not After"

方法三：公司 CA 憑證（企業環境）

企業

公司通常會提供 .crt（憑證）和 .key（私鑰）檔案，直接上傳到伺服器使用。

bash
## 上傳憑證（從 Windows 用 scp） PS> scp myapp.crt myapp.key admin@192.168.50.130:/tmp/ ## 移動到正確位置 $ sudo mkdir -p /etc/nginx/ssl $ sudo mv /tmp/myapp.crt /tmp/myapp.key /etc/nginx/ssl/ $ sudo chmod 600 /etc/nginx/ssl/myapp.key # 私鑰只有 root 能讀

Nginx 掛 HTTPS 設定

設定
/etc/nginx/sites-available/myapp
# HTTP → HTTPS 跳轉 server { listen 80; server_name myapp.example.com; return 301 https://$host$request_uri; } # HTTPS 主設定 server { listen 443 ssl; server_name myapp.example.com; # 憑證路徑 ssl_certificate /etc/nginx/ssl/myapp.crt; ssl_certificate_key /etc/nginx/ssl/myapp.key; # SSL 安全設定 ssl_protocols TLSv1.2 TLSv1.3; ssl_ciphers HIGH:!aNULL:!MD5; # 反向代理到 Tomcat location / { proxy_pass http://127.0.0.1:8080; proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; proxy_set_header X-Forwarded-Proto https; } }

Web Server vs AP Server 架構

架構概念 Nginx + Tomcat
理解 Nginx（Web Server）和 Tomcat（AP Server）的分工，是設計伺服器架構的基礎。

完整架構圖

架構
Internet
│ Port 80 / 443
▼
🌐 Nginx（Web Server）
✅ 處理 HTTPS / SSL 憑證
✅ 服務靜態檔案（快）
✅ 負載均衡分流
✅ 請求壓縮、快取
│ HTTP 內部 Port 8080
▼
🐱 Tomcat（AP Server）
✅ 執行 Java / Spring Boot
✅ 處理業務邏輯
✅ Session 管理
❌ 不直接對外
│ JDBC Port 5432 / 1521
▼

PostgreSQL / Oracle

各自負責什麼？

分工
🌐 Nginx (Web Server)🐱 Tomcat (AP Server)
對外 Port80（HTTP）/ 443（HTTPS）8080（僅內部）
SSL✅ 在這裡終止 SSL不需要處理
靜態檔✅ 直接回應（速度快）也可以，但沒 Nginx 快
動態請求轉發給 Tomcat✅ Java 執行業務邏輯
負載均衡✅ 分流多台 Tomcat—
壓縮/快取✅ gzip 回應壓縮—
連線數上限可處理數萬並發受 JVM 限制
💡
為什麼要分開？Nginx 用 C 語言寫的，處理大量 HTTP 連線效率極高。Tomcat 用 Java，適合業務邏輯但不適合直接對大量連線。兩者搭配各取所長。

Load Balance / Session 處理

高可用 進階
多台 Tomcat 時，需要考慮負載均衡和 Session 問題。這是企業環境的重要架構概念。

問題：多台 Tomcat 的 Session 困境

問題
使用者 第1次 → Nginx → Tomcat-A ✅ 登入成功，Session 存在 A
使用者 第2次 → Nginx → Tomcat-B ❌ 找不到 Session！要求重新登入
⚠️ 原因：Session 預設儲存在各自 Tomcat 記憶體，不共享

解法一：Session Sticky（IP Hash）

解法一

讓同一個使用者的請求，永遠送到同一台 Tomcat。

/etc/nginx/sites-available/myapp
upstream tomcat_cluster { ip_hash; # 同一個 IP 永遠送同一台 server 192.168.50.131:8080; server 192.168.50.132:8080; } server { location / { proxy_pass http://tomcat_cluster; } }
✅
優點：設定簡單，不需要修改應用程式
⚠️
缺點：那台 Tomcat 掛了，Session 就消失，使用者需重新登入

解法二：Session 共享（Redis）— 建議

最佳解
使用者
↓
Nginx（輪詢，任何一台都行）
↓
Tomcat-A
↕ 讀/寫 Session
Tomcat-B
↕ 讀/寫 Session
↕
Redis（Session 集中儲存）
✅ 不管哪台 Tomcat 回應，都從 Redis 取 Session
✅ 任何一台 Tomcat 掛了，Session 不會消失

Spring Boot 設定 Redis Session

pom.xml
<dependency> <groupId>org.springframework.session</groupId> <artifactId>spring-session-data-redis</artifactId> </dependency> <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-data-redis</artifactId> </dependency>
application.properties
spring.session.store-type=redis spring.data.redis.host=192.168.50.133 spring.data.redis.port=6379 server.servlet.session.timeout=30m
💡
加上 @EnableRedisHttpSession annotation 在主 class 上，就完成了！

Nginx 負載均衡完整設定

設定
/etc/nginx/sites-available/myapp
upstream tomcat_pool { # 輪詢（預設），搭配 weight 可加權 server 192.168.50.131:8080 weight=2; # 這台效能好，多分流 server 192.168.50.132:8080 weight=1; server 192.168.50.133:8080 backup; # 備援，正常不用 # 健康檢查：連不上超過 3 次排除 30 秒 max_fails=3 fail_timeout=30s; } server { listen 443 ssl; ... location / { proxy_pass http://tomcat_pool; # 把真實 IP 傳給後端 proxy_set_header X-Real-IP $remote_addr; proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; } }
